# Stock Exchange Order Matching Engine Simulator

A simulated stock exchange with a real, working price-time-priority **matching engine**, a Django/PostgreSQL backend, a React dashboard, and a swarm of trading **bots** that keep the market moving so there's always something to match against.

This is a learning/portfolio project, not a production trading system ‚Äî see [Known issues & limitations](#known-issues--limitations) before you point it at anything real.

## Overview

There's no real-world market data feed here ‚Äî instead, autonomous bot accounts ("market makers" and "news traders") constantly place orders against each other, optionally reacting to admin-injected "news" events. Every order, real or bot-placed, goes through the exact same pipeline: it's saved to Postgres, a Django signal hands it to Celery, and a `MatchingEngine` service tries to cross it against the resting order book using standard price-time priority.

## Features

- Continuous double-auction matching engine (price-time priority, maker-price execution)
- Asynchronous order processing via Django signals ‚Üí Celery ‚Üí Redis
- Per-stock order book with live bid/ask depth
- Full audit trail: every fill creates an immutable `Transaction` record
- User balances and per-stock portfolios that update automatically on every trade
- Two bot strategies (market maker, news trader) plus a scheduled simulation heartbeat
- "God Mode" market events that bias bot sentiment by industry (or globally)
- React dashboard: market watch, live order book, participant leaderboard, event controls
- REST API with auto-generated Swagger/Redoc docs (drf-spectacular)
- Dockerized: one `docker-compose.yaml` runs Postgres, Redis, Django, two Celery processes, and the frontend

## Architecture

### Order lifecycle

Every order ‚Äî whether typed into the API by a human or generated by a bot ‚Äî flows through the same pipeline:

```
 Order placed (human via API, or a bot)
        ‚îÇ  Order.objects.create(status=OPEN)
        ‚ñº
 Django post_save signal fires
        ‚îÇ  transaction.on_commit(...) ‚Üí process_order_task.delay(order_id)
        ‚ñº
 Celery task queued via Redis
        ‚ñº
 MatchingEngine.process_order()
        ‚îÇ  OrderBook.get_best_match(order)
        ‚ñº
   match found? ‚îÄ‚îÄNo‚îÄ‚îÄ‚ñ∂ order rests in the book (status stays OPEN)
        ‚îÇ Yes
        ‚ñº
 _execute_trade()
   - debit buyer / credit seller balance
   - update both portfolios
   - create a Transaction record
   - update Stock.spot_price and volume
   - mark order(s) PARTIAL or FILLED
```

The engine loops on the *incoming* order until it's fully filled or no more crossable orders remain, locking rows with `select_for_update()` as it goes to avoid two workers double-filling the same order.

### Bot simulation loop

A Celery beat schedule fires every 5 seconds and runs all active bots, whose orders feed into the exact same pipeline above:

```
 Celery beat (every 5s) ‚Üí apps.simulation.tasks.run_simulation_cycle
        ‚îÇ
        ‚îú‚îÄ‚îÄ Market maker bots
        ‚îÇ     - delete their own stale OPEN orders
        ‚îÇ     - quote a tight bid/ask around the stock's spot price
        ‚îÇ
        ‚îî‚îÄ‚îÄ News trader bots
              - read active MarketEvent rows for their stock's industry
              - shift buy/sell probability by each event's sentiment_score
              - place a BUY/SELL order, or hold
        ‚îÇ
        ‚ñº
 Order.objects.create(...) ‚Üí feeds into the order lifecycle above
```

## Core concepts

### The matching engine (`services/matching_engine.py`)

`MatchingEngine.process_order(order_id)` re-fetches the order fresh from the DB (never trusts an in-memory object), then repeatedly:

1. Asks `OrderBook.get_best_match()` for the best opposing order.
2. Confirms the prices actually cross (a buy's limit price ‚â• the best ask, or a sell's limit price ‚â§ the best bid).
3. Executes at the **resting (maker) order's price**, for `min(remaining quantities)` of the two orders.
4. Updates balances, portfolios, the stock's `spot_price`/`volume`, writes a `Transaction`, and updates both orders' status.
5. Repeats until the incoming order is filled or nothing left can match ‚Äî at which point it simply rests in the book.

### The order book (`services/order_book.py`)

Not an in-memory structure ‚Äî it's two simple, indexed queries against the `Order` table:

```python
def get_bids(self):
    return Order.objects.filter(
        stock=self.stock, side='BUY', status__in=['OPEN', 'PARTIAL']
    ).order_by('-price', 'created_at')   # best price first, oldest first on ties
```

That `order_by('-price', 'created_at')` (and its mirror for asks) *is* the price-time priority rule ‚Äî same idea every real exchange uses, just expressed as a DB query instead of a heap.

### Bots & strategies (`apps/simulation/`)

- **Market maker** (`strategies/market_maker.py`): every cycle, cancels its own stale quotes and places a fresh bid/ask pair straddling the stock's spot price with a tight ~1% spread. Exists purely to keep the book liquid.
- **News trader** (`strategies/news_trader.py`): starts from a neutral 50/50 buy/sell sentiment, shifts it by any active `MarketEvent`'s `sentiment_score` for that stock's industry (or `GLOBAL`), adds a little random noise scaled by the bot's own `risk_tolerance`, and either places an order anchored near the last trade price or holds.

### Market events ‚Äî "God Mode" (`apps/events/`)

A `MarketEvent` is just `{title, target_industry, sentiment_score, is_active}`. Creating one (via the API, or the dashboard's "God Mode" panel) is the only lever you need to make the whole simulated market lean bullish or bearish on a sector ‚Äî every news trader bot picks it up on its next decision.

## Getting started

### Prerequisites

- **Docker** and **Docker Compose** (recommended path), **or**
- Python **3.12+** (the Dockerfile uses 3.13; Django 6.0 requires 3.12+)
- Node.js **18+** and npm
- A running **PostgreSQL** server
- A running **Redis** server

### Option A ‚Äî Docker Compose (recommended)

1. Create `backend/.env` (docker-compose reads it; values just need to match `DB_*`/`POSTGRES_*` to each other):

   ```env
   POSTGRES_DB=exchange_db_v1
   POSTGRES_USER=emirhan
   POSTGRES_PASSWORD=changeme

   DB_NAME=exchange_db_v1
   DB_USER=emirhan
   DB_PASSWORD=changeme
   ```

2. Build and start everything:

   ```bash
   cd backend
   docker compose up --build
   ```

   This brings up five services: `web` (Django, :8000), `celery` (worker), `celery-beat` (scheduler), `db` (Postgres, :5432), `redis`, plus `frontend` (React dev server, :3000).

3. In a second terminal, run migrations and create an admin user:

   ```bash
   docker compose exec web python manage.py migrate
   docker compose exec web python manage.py createsuperuser
   ```

4. Seed the market with stocks and bots:

   ```bash
   docker compose exec web python manage.py reset_simulation
   ```

5. **Before the frontend will compile**, fix the import bug described in [Known issues](#known-issues--limitations). Then open:
   - Dashboard ‚Äî http://localhost:3000
   - API root ‚Äî http://localhost:8000/api/
   - Swagger UI ‚Äî http://localhost:8000/api/schema/swagger-ui/
   - Django admin ‚Äî http://localhost:8000/admin/

### Option B ‚Äî Manual setup (no Docker)

**Backend:**

```bash
cd backend
python -m venv venv
source venv/bin/activate        # Windows: venv\Scripts\activate
pip install -r requirements.txt

# With Postgres and Redis already running locally:
export DB_NAME=exchange_db_v1 DB_USER=emirhan DB_PASSWORD=changeme DB_HOST=localhost
export CELERY_BROKER_URL=redis://localhost:6379/0
export CELERY_RESULT_BACKEND=redis://localhost:6379/0

python manage.py migrate
python manage.py createsuperuser
python manage.py reset_simulation
python manage.py runserver
```

In two more terminals (same venv activated):

```bash
celery -A config worker -l info
celery -A config beat -l info
```

Without the worker, orders will sit at `OPEN` forever ‚Äî nothing ever calls the matching engine. Without beat, no bots trade and the book stays empty.

**Frontend:**

```bash
cd frontend
npm install
npm start
```

Runs on http://localhost:3000 and talks to `http://localhost:8000/api/` ‚Äî hardcoded in `src/api.js`, so the backend must be reachable at that exact URL.

## Seeding & resetting data

| Command | What it does |
|---|---|
| `python manage.py init_bots` | Creates 5 market-maker bots ($1,000,000 balance each) and 20 news-trader bots ($50,000 each, random risk tolerance). Idempotent ‚Äî safe to re-run. |
| `python manage.py reset_simulation` | **Factory reset.** Wipes transactions, orders, portfolios, events, bots, non-superuser users, and stocks; recreates 6 stocks (TSLA, AAPL, NVDA, XOM, JPM, PFE) across Tech/Energy/Finance/Health; re-runs `init_bots`. Run this any time you want a clean slate. |
| `python manage.py reset_small` | A minimal debug scenario: one stock (TSLA), one market-maker "whale" holding 1,000 shares, two aggressive news-trader bots, and one large bullish `MarketEvent` already active ‚Äî so you can watch a trade happen within seconds without waiting on the wider simulation. |

## Using the simulator

The dashboard is three columns:

1. **Market Watch** ‚Äî every stock and its live spot price (polls every 3s). Click one to select it.
2. **Order Book** ‚Äî live bid/ask depth for the selected stock (polls every 2s), asks on top, bids below the spread.
3. **Market Participants / God Mode** ‚Äî a leaderboard of every user/bot's cash balance (polls every 5s), and three preset buttons to fire a `MarketEvent` (rate cut, tech crash, oil boom) and watch the news-trader bots react within the next 5-second beat tick.

The dashboard itself doesn't expose an order-entry form ‚Äî place orders via the API or Swagger UI (see below).

```bash
        "user": "Alice",
        "stock": "TSLA",
        "side": "BUY",
        "price": "200.00",
        "quantity": 10
      }'
```

**Triggering a market event:**

```bash
curl -X POST http://localhost:8000/api/events/ \
  -H "Content-Type: application/json" \

        "is_active": true
      }'
```

Valid `target_industry` values: `Technology`, `Health`, `Finance`, `Energy`, `ConsumerGoods`, or `GLOBAL`.

## Running the tests

The three root-level scripts under `backend/` are standalone debug scripts (they call `django.setup()` and print pass/fail expectations to the console) rather than a pytest or `TestCase` suite ‚Äî every `apps/*/tests.py` is still the empty Django-generated placeholder, so `python manage.py test` currently passes trivially with no real coverage.

```bash
python test_engine.py            # Places a SELL + BUY, calls MatchingEngine directly ‚Äî synchronous, no Celery needed
python test_automated.py         # Same scenario, but via the real signal ‚Üí Celery ‚Üí Redis pipeline (worker must be running)
python test_simulation_logic.py  # Sanity-checks the market-maker/news-trader strategy pricing logic in isolation
```

## Known issues & limitations

- **Frontend won't compile as-is**: `App.js` imports `EventControls` from `./components/EventControls`, but the file on disk is `components/EventControl.js` (no trailing "s"). On a case-sensitive filesystem ‚Äî Linux, macOS, and inside the Docker image ‚Äî this is a hard "module not found" error. Fix it by renaming the file to `EventControls.js` or changing the import to `./components/EventControl`.
- **The API is open by default**: JWT and allauth are installed and configured, but `REST_FRAMEWORK['DEFAULT_PERMISSION_CLASSES']` is empty, so every endpoint except `/api/users/me/` works with no token at all. Fine for a local simulation, not something to expose publicly unmodified.
- **No balance or inventory checks**: `_execute_trade()` debits the buyer's balance and the seller's shares unconditionally ‚Äî nothing stops either from going negative. There's no margin or short-selling logic; it's simply not enforced.
- **Orders can't be cancelled via the API**: the `Order` model has a `CANCELLED` status, but the only route is `GET`/`POST` on `/api/orders/` ‚Äî no `PATCH`/`DELETE` to cancel a resting order yet.
- **Django admin is unconfigured**: every `apps/*/admin.py` is the default stub with nothing registered, so `/admin/` is only useful for logging in as a superuser right now, not for browsing data.
- **Secrets are committed**: `SECRET_KEY` is a hardcoded string and `DEBUG = True` is unconditional in `config/settings.py`. Both need to change before this runs anywhere but localhost.
- **Runtime artifacts are checked into git**: `celerybeat-schedule-shm`, `celerybeat-schedule-wal`, and `dump.rdb` (a Redis snapshot) are committed under `backend/`. These regenerate automatically ‚Äî consider adding `celerybeat-schedule*` and `dump.rdb` to `.gitignore` and removing them from version control.
- **No `.env.example`**: `docker-compose.yaml` expects `backend/.env`, but no template ships in the repo ‚Äî use the [Getting Started](#getting-started) values above as a starting point.

## Possible next steps

- Add `PATCH`/`DELETE` on `/api/orders/<id>/` so resting orders can actually be cancelled
